Otto docs are structured for operators first. The goal is to answer three
questions quickly:

1. What behavior is expected right now?
2. Which contract defines that behavior?
3. What do I do if behavior drifts?

Use the top-level sections based on your task:

- **Concepts** for shared runtime language and architecture context.
- **Contracts** for guarantees, boundaries, and compatibility expectations.
- **Operator Guide** for workflows, runbooks, and recovery procedures.
- **CLI Reference** for `ottoctl` command-level behavior.
- **API Reference** for external API semantics and generated contract sources.

For contribution standards, see [Contributing](./contributing.md).
